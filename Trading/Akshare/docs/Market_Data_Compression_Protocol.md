# 极致行情压缩协议 (V6 终极版) 技术文档

本协议设计用于在 Telegram 消息的 4096 字符限制内，传输大规模 (50+ 板块, 200+ 个股) 的精细行情数据。通过二进制编码、位运算和静态字典映射，实现了极高的压减率。

---

## 1. 字典说明 (Metatada)

为了消灭消息中的重复中文字符，本协议依赖一套**静态板块字典**。接收端（HTML）需预载此字典以还原信息。

### 1.1 板块字典 (`concept_dict.json`)
*   **用途**: 将变长的中文板块名映射为固定长度的 ID。
*   **格式**: `{ "人工智能": 0, "半导体": 1, ... }`
*   **存储**: 传输时使用 **2 字节 (Uint16)** 表示 ID，支持最高 65,535 个板块。当前考察范围约 329 个。

### 1.2 股票代码 (不使用字典)
*   **逻辑**: 考虑到全市场股票代码均为 6 位数字，且变动较快，本协议选择**直接传输 6 位数字代码**。
*   **规则**: 接收端提取数字后，根据首位规律 (6/0/3/8) 自动补全交易所前缀 (sh/sz/bj) 以调用第三方实时行情。

---

## 2. 消息包二进制结构 (Binary Structure)

完整的消息由 **二进制包头** + **板块数据区** + **股票数据区** 组成，最后经 Zlib 压缩并转为 Base64。

### 2.1 基础头部 (Header) - 7 Bytes
| 字段 | 长度 | 类型 | 分位说明 |
| :--- | :--- | :--- | :--- |
| Magic | 2 字节 | Char | 固定为 `M6` |
| Version | 1 字节 | Uint8 | `0x06` |
| Timestamp | 4 字节 | Uint32 | **高15位**: 距 2026-01-01 天数 <br> **低17位**: 当日秒数 (0-86399) |

*Timestamp 说明*: 采用结构化存储，支持约 90 年的时间跨度且具备秒级精度。前端可直接通过位运算提取日期和时间，无需复杂的 Date 对象计算。

### 2.2 数据分区结构 (4 Sections)
协议严格划分为 4 个独立数据区，每个区头部携带 2 字节数量标识。

#### A. 核心异动板块 (Primary Sectors)
*   **内容**: 包含所有 **Pool 1 (活跃), Pool 2 (强势), Pool 3 (爆发)** 的新增板块去重合集。
*   **Count**: 2 字节 (Uint16)
*   **Item (13 Bytes)**:
    - ID (2B) + 涨跌幅 (2B) + 涨停数 (1B) + 跌停数 (1B) + 上涨家数 (2B) + 下跌家数 (2B) + 换手率 (2B) + **Pool Level (1B)**
    - *Pool Level 定义*: 1=活跃 (P1), 2=强势 (P2), 3=爆发 (P3)

#### B. 预留板块区 (Secondary Sectors)
*   **内容**: 当前置空 (Count=0)，预留给未来扩展或从属板块。
*   **Count**: 2 字节 (Uint16)
*   **Item (13 Bytes)**: 同上。

#### C. 冲板捕捉股票 (Lottery Pool)
*   **Count**: 2 字节 (Uint16)
*   **Item (变长)**:
    - **Code (3B)**: Uint24 (6位数字)
    - **Flag (1B)**: Bit 7 必须为 1
    - **Concept Count (1B)**: Uint8
    - **Concept IDs (N*2B)**: 关联的板块 ID 列表 (对应 `concept_dict.json` 索引)

#### D. 小票池股票 (Small Cap Pool)
*   **Count**: 2 字节 (Uint16)
*   **Item (变长)**:
    - **Code (3B)**: Uint24 (6位数字)
    - **Flag (1B)**: Bit 6 必须为 1 (若同时冲板则 Bit 7 也为 1)
    - **Concept Count (1B)**: Uint8
    - **Concept IDs (N*2B)**: 关联的板块 ID 列表 (对应 `concept_dict.json` 索引)

*(注: 现价、涨跌幅等基础行情数据已移除，由接收端根据代码调用第三方 API 实时获取)*

---

## 3. 股票标志位 (Flag Byte) 深度定义

使用单字节位图 (Bitmask) 标识股票的推荐逻辑与异动原因，支持多重标准叠加。

| 位 (Bit) | 标识含义 | 逻辑说明 |
| :--- | :--- | :--- |
| **Bit 7** | **冲板捕捉** | 1: 符合系统实时冲板捕捉算法 (Part 3) |
| **Bit 6** | **小票池推荐** | 1: 该股票命中离线知识库的小票池映射 (Part 4) |
| **Bit 5-0** | **预留位** | 默认为 0，预留给未来更细分的异动原因代码。 |

## 4. 极端场景容量评估

| 场景规模 | 原始字节 | Zlib 压缩后 | Base64 长度 | Telegram 状态 |
| :--- | :--- | :--- | :--- | :--- |
| **20板 + 50股** | 约 900B | 约 500B | 约 670 字符 | **极度空闲** |
| **50板 + 200股** | 约 3200B | 约 2000B | 约 2660 字符 | **安全运行** |
| **100板 + 300股** | 约 5800B | 约 3500B | 约 4660 字符 | **上限边缘** |

## 5. 传输与还原流程

1.  **数据脱水**: 服务端根据上述协议，将 Python 对象序列化为二进制 Buffer。
2.  **终极压缩**: 对 Buffer 执行 Zlib (Level 9) 压缩。
3.  **封包**: 执行 URL 安全的 Base64 编码，构造链接 `https://your-ui.com/?d=BASE64_STR`。
4.  **接收解包**: 前端读取 `d` 参数 -> Base64 解码 -> Zlib 解压 -> DataView 逐字节读取 -> 查表还原中文。
5.  **异步增强**: 前端根据代码异步调用第三方接口，补全 K 线、成交额、总市值等重数据。
